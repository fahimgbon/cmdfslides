Part 1

Does Anyone Know What a PRD Is?

PRD Definition

Product Requirements Document

🧭

WHAT we're building

The specific product outcome, clearly named and described.

👤

FOR WHOM

The exact user segment this project is designed for.

✨

WHY it matters

The core value and reason this should exist right now.

Without a PRD

Everyone builds in different directions.

Frontend

Builds a full settings page.

❌ No agreed behavior

Backend

Builds no settings endpoint.

❌ Missing contract

Design

Designed 3 screens nobody asked for.

❌ No shared target

With a PRD

Everyone ships to the same spec.

Frontend

Builds only the approved settings UI.

✅ Scoped to requirements

Backend

Implements the matching endpoint.

✅ Contract-first delivery

Design

Designs only what supports the same flow.

✅ Aligned output

Part 2

Who Are We Building For?

What Is A User Story
As a [user], I want [feature] so that [outcome].

Human-readable. Outcome-driven. Easy for product, design, and engineering to align on.

Good vs Bad
BAD

“As someone, I want the timer to work so things happen.”

GOOD

“As a student, I want to be notified when my focus session ends so I can take a break.”

Pomodoro Examples

4 user stories from the app

🔔 Get a notification when focus ends
▶️ Start, pause, and reset sessions
🍓 Earn random rewards per completed session
🖼️ View collected fruits in a gallery

Part 3

What Are We NOT Building?

Scope is a superpower.

Scope Creep

Definition

Scope creep is when “small additions” gradually turn a focused project into an impossible one.

Relatable example: a timer app slowly becomes a social network, task manager, and analytics dashboard.

↻
MoSCoW Method

Must / Should / Could / Won’t

MUST HAVE (gold)
SHOULD HAVE (silver)
COULD HAVE (teal)
WON'T HAVE THIS TIME (grey)

Part 4

Why API Alignment Matters for Your PRD

This is not a backend lesson. This is a clarity lesson. The API section in your PRD is how your team avoids “I thought you were building that.”

Connection Flow

PRD line → User story → API handshake → Working feature

PRD

“Students can save timer preferences.”

User Story

As a student, I want my timer saved so every session starts how I like it.

API Handshake

POST /settings with timer length, returns saved state.

When this chain is explicit, you ship faster with less rework.

What Goes In The PRD

The 4-line API handshake your team needs

1. Purpose: what user moment this supports

2. Input: what data is sent

3. Success response: what comes back

4. Failure behavior: what happens if it fails

Handshake Example

POST /settings
body: { sessionLength: 25 }
returns: { saved: true, sessionLength: 25 }
on error: show retry toast

Build-Readiness Check

Before coding, confirm these 5 things

✅ We agree on the target user

✅ We agree on must-have features

✅ We agree on what we are cutting

✅ We agree on key API handshakes

✅ We agree on what “done” looks like

If any are fuzzy, pause and clarify now.
